================================================================================
  UPHOLSTERY! — GAME DESIGN DOCUMENT
  Original game + New Physics-Based Concept
  Written: 2026-06-29
================================================================================

This document covers two things:
  A) The current UPHOLSTERY! game (already built, in production)
  B) A new physics-based concept game (to be prototyped in a separate Roblox place)

Read both sections before starting work. The concept game draws on lessons from the
original and shares its thematic identity, but is mechanically distinct.

================================================================================
  SECTION A — CURRENT UPHOLSTERY! (REFERENCE)
================================================================================

--------------------------------------------------------------------------------
  A1. OVERVIEW
--------------------------------------------------------------------------------

UPHOLSTERY! is a Roblox job simulator where players run an upholstery repair
business. They browse job listings on an in-game laptop, accept jobs, teleport to
client locations, complete repair minigames, and return to their personal house.
The social layer (Talkogram) posts their results publicly. The economy tracks
coins (USD), followers, and reputation.

The game is built with Rojo (file-based sync to Roblox Studio). All code lives
under src/ split into server/, client (gui/), and shared/.

--------------------------------------------------------------------------------
  A2. CORE GAMEPLAY LOOP
--------------------------------------------------------------------------------

1. Player opens laptop via proximity prompt at their house
2. Laptop boots (animated progress bar), shows desktop OS
3. Player navigates to MoneyMaker tab — a job board
4. Job board shows a pool of procedurally generated jobs (4-12 entries)
5. Player clicks Accept on a job → laptop closes
6. Server teleports player to a client location (instanced slot, cloned on demand)
7. Player walks to furniture and activates a ProximityPrompt
8. One or more minigames play in sequence (GUI overlay)
9. Player exits via a door ProximityPrompt → result submitted
10. Player teleports back to their personal house
11. Talkogram auto-posts the result to the social feed
12. Pool refreshes on a timer; repeat

--------------------------------------------------------------------------------
  A3. MINIGAMES (CURRENT — 4 TOTAL)
--------------------------------------------------------------------------------

All 4 minigames are GUI-based overlays. They run inside a blurred screen frame.
The engine is MinigameRunner.luau which handles blur, result overlay, and chain
execution. Each minigame module exposes: Start(container, config), GetResult(),
Cleanup(), Cancel().

  TACK REMOVAL (TackRemoval.luau)
    Style: osu!-style rhythm. Circular tack icons appear with an approach ring.
    Player clicks as the ring reaches the tack. Timing determines Perfect/Good/Miss.
    Config: tackCountMin/Max, approachTime, perfectMinProgress, goodMinProgress.

  STAPLE GUNNING (StapleGunning.luau)
    Style: Moving bar + fire button. Bar oscillates across zones.
    Player presses button when bar is in the Perfect or Good zone.
    Config: stapleCountMin/Max, barSpeed, goodZoneWidth, perfectZoneWidth,
            barSpeedRampUp, timeLimit.

  SEAM STITCHING (SeamStitching.luau)
    Style: Free-drag needle along a dotted path. No checkpoints — one continuous
    motion from start marker to end marker.
    Deviation is measured against the NEAREST point on the entire path.
    A miss-budget bar fills as the needle strays off-path. Reaching the end while
    under budget grades by how full the budget got.
    Config: segmentCountMin/Max, timePerSegment, deviationTolerance (fixed),
            missBudget (primary difficulty lever: Easy=4.5, Medium=2.5, Hard=1.2),
            perfectMaxDeviationFrac, goodMaxDeviationFrac.
    Result: Fail (didn't reach end or budget exhausted), Pass, Good, Perfect.

  FOAM PADDING (FoamPadding.luau)
    Style: Drag-and-drop puzzle. Irregular mold shapes must be filled by placing
    tetromino-like foam pieces onto a grid. Pieces are generated with adjacency-
    biased placement (connected mold region, never disconnected islands).
    Config: pieceCount, timeLimit, perfectMinRatio, goodMinRatio.

All minigames support a difficulty preset system (Easy/Medium/Hard) defined in
GameConfig.luau under DifficultyPresets. A Fragile modifier tightens tolerance
windows by a multiplier (fragileMultiplier = 0.6).

--------------------------------------------------------------------------------
  A4. JOB SYSTEM
--------------------------------------------------------------------------------

Jobs are procedurally generated by RadiantJobGenerator.luau on the server.
Each job has:
  - name: furniture name + damage description
  - tier: Starter / Mid / Premium / Prestige / Exclusive / Elite / Special
  - pay: coins, scaled by tier range (GameConfig.Economy.payByTier)
  - followers: social reward, scaled by tier
  - difficulty: Easy / Medium / Hard (maps to DifficultyPresets)
  - fragile: bool (true for Premium and above)
  - minigames: ordered array of minigame names for the chain
  - furniture: array of furniture model keys (one per piece)

Job tier rarity on the board is weighted (Starter=50, Mid=28, Premium=13,
Prestige=6, Exclusive=2, Elite=0.7, Special=0.3).

Jobs are first-come-first-served. Accepting a job removes it from the shared pool.
The pool refreshes every 15-30 seconds.

Hot Jobs: Every 5-8 minutes, a special high-pay job (3x multiplier, +75%
followers) appears with a 10-second countdown warning.

--------------------------------------------------------------------------------
  A5. ECONOMY
--------------------------------------------------------------------------------

  USD (coins):    Earned on job completion. Spent in the shop on perks.
  Followers:      Earned on job completion. Used as a social/reputation metric.
  Reputation:     0.0–5.0 float. Rises with good results, falls on Fail/Abandon.
                  Unlocks perk toolbox slots at 1.5 (4th slot) and 3.0 (5th slot).

Reputation gain/loss per result:
  Perfect: +0.20 | Good: +0.10 | Pass: +0.05
  Fail: scaled by tier (Starter -0.03 → Elite -0.30)
  Abandon: scaled by tier (Starter -0.05 → Elite -0.45)

All stats persist via DataStore (PlayerData_v1).

--------------------------------------------------------------------------------
  A6. PERK SYSTEM (TOOLBOX)
--------------------------------------------------------------------------------

Players buy bolt perks from the Shop tab on the laptop. Perks have charges (uses).
Each perk has an effect that modifies minigame parameters for one run.

Perk equip slots: 3 base, up to 5 unlocked by reputation milestones.
Charges burn after each job run. Failing a job burns an extra charge (compound
failure cost). Perks with 0 charges auto-unequip.

Effect examples:
  curvedNeedle: deviationToleranceBonus for SeamStitching
  foamPressKit: snapToleranceBonus for FoamPadding
  tackMagnet: approachTimeBonus for TackRemoval
  stapleAssist: zoneWidthMultiplier for StapleGunning
  payBooster: payMultiplier on job reward
  followerBooster: followerMultiplier on job reward

All perk data lives in ShopData.luau. PerkSystem.luau manages server-side equip,
charge tracking, GetActiveEffects(), BurnCharges(). MinigameRunner applies effects
via buildParams().

--------------------------------------------------------------------------------
  A7. LOCATION SYSTEM
--------------------------------------------------------------------------------

Client locations are slot templates stored in ServerStorage. On job accept, a slot
is cloned into Workspace/ActiveSlots for the duration of the job, then destroyed on
release. Max 6 simultaneous players per server (6 house slots).

Slot templates contain: Floor, Walls folder, SpawnPad (player spawn), FurniturePad_N
(furniture placement positions), Door (with DoorPrompt ProximityPrompt).

Furniture models are in ServerStorage/FurnitureModels/[Key]/Damaged_A/B/C and
Default. On assign: a random Damaged variant is cloned in with randomized fabric
and frame colors. On repair: Damaged clone is destroyed, Default clone placed.

Repair prompts are all active simultaneously (free order). Only one prompt shows
at a time (MaxPromptsVisible = 1, nearest wins).

Door prompt triggers job submission. If mid-minigame, it cancels the minigame
first (MinigameRunner.Cancel()), then submits.

--------------------------------------------------------------------------------
  A8. SOCIAL — TALKOGRAM
--------------------------------------------------------------------------------

Talkogram is an in-game social feed resembling a simplified Instagram/Twitter.
After every successful job, a post is auto-generated with:
  - A title based on result tier (Perfect/Good/Pass)
  - The furniture/job name
  - The player's username and avatar

Posts are stored server-side and visible to all players on the same server.
Players can like and comment on posts. Talkogram is a tab on the laptop.

--------------------------------------------------------------------------------
  A9. PLAYER HOUSES
--------------------------------------------------------------------------------

Each player gets a personal house slot (House_1 through House_6 in Workspace).
Players spawn/return to their own house, not a shared hub. The house contains
their laptop (the main interaction point for everything).

Future: House prestige upgrades at follower milestones (2k, 10k, 50k) swap the
house model for a nicer one (CornerHouse, WorkshopVilla, SkylinePenthouse).

--------------------------------------------------------------------------------
  A10. WHY THE ORIGINAL GAME NEEDS RETHINKING (CONTEXT FOR SECTION B)
--------------------------------------------------------------------------------

The original game's GUI minigames are functionally solid but feel disconnected from
the upholstery theme. Players are doing abstract rhythm/puzzle games rather than
feeling like they're actually fixing furniture. The game is also low on spatial
exploration — you teleport in, interact, teleport out. There is no sense of physical
presence at the job site. The new concept (Section B) addresses both of these.

--------------------------------------------------------------------------------

--------------------------------------------------------------------------------

The following models already exist in the concept place. Do NOT replace them

  SOFA:

    "SofaClean"     — the finished sofa after full repair (used for State 4 model swap)

    The Cushion part is the snap zone for foam and cloth placement.

  TOOLBOX:

  TOOLS (all need "Grabbable" CollectionService tag):

    "StapleGun"     — triggers StapleGunning GUI when held near State 3 sofa

  MATERIALS (all need "Grabbable" CollectionService tag):
    "Sponge"        — the foam padding piece (snaps to State 1 sofa cushion)
    "Cloth"         — the fabric piece (snaps to State 2 sofa cushion)

  LOCATION:
    The repair room/location already exists in the place. It has a front door
    where the NPC door interaction (see B5a) will be placed.

  IMPORTANT NAMING: In all code, use these exact model names. When the design
  document says "foam piece" it means the "Sponge" model. "Cloth/fabric" means
  "Cloth". "Needle" means "SewingNeedle".

--------------------------------------------------------------------------------
  B1. CONCEPT OVERVIEW
--------------------------------------------------------------------------------

This is a separate Roblox place — a prototype/concept, not a direct replacement
for the original game yet. It reimagines the repair workflow using a physics-based
grabbing system. Players physically carry, throw, and place materials at job sites.
The abstract GUI minigames are mostly replaced by real spatial interactions.

The design intent is similar to "A Dusty Trip" or "The Long Drive" — games where
the physicality of objects and the risk of mishandling creates emergent tension and
fun. A player could throw foam padding across the room and have it snap perfectly
onto the sofa. That's more interesting than clicking a button.

The grabbing system (ConceptGrabbingSystem/) already exists as production-quality
code. It handles: server-authoritative grab validation, cooldowns, mass limits,
collision group isolation (held objects don't collide with the player), network
ownership transfer, and throw with mouse-aim direction.

--------------------------------------------------------------------------------
  B2. THE GRABBING SYSTEM — WHAT ALREADY EXISTS
--------------------------------------------------------------------------------

Files:
  ConceptGrabbingSystem/shared/GrabConfig.luau  — all tunable constants
  ConceptGrabbingSystem/shared/GrabRemotes.luau — creates/fetches RemoteEvents
  ConceptGrabbingSystem/client/GrabController.luau — hover highlight, grab, throw
  ConceptGrabbingSystem/client/init.client.luau  — entry point
  ConceptGrabbingSystem/server/GrabHandler.luau  — validation, ownership, cleanup
  ConceptGrabbingSystem/server/init.server.luau  — entry point

Key constants (GrabConfig.luau):
  GRAB_KEY = E           — grab / release
  THROW_KEY = F          — throw (also left-click while holding)
  MAX_GRAB_DISTANCE = 12 studs
  MAX_OBJECT_MASS = 5000
  CARRY_DISTANCE = 5 studs in front of player
  CARRY_SPEED = 20, MAX_CARRY_SPEED = 50
  THROW_SPEED = 40, THROW_UP_BIAS = 0.15
  GRAB_COOLDOWN = 0.3s
  GRABBABLE_TAG = "Grabbable" (CollectionService tag on any grabbable part/model)

How grabbing works (summary):
  Client raycasts from camera through mouse position. If hit part is within range
  and has the "Grabbable" tag (or a Grabbable-tagged ancestor), client fires
  RequestGrab to server. Server validates (not already held by another, not too
  heavy, player alive, in range), unanchors the object, swaps it to "HeldObjects"
  collision group (doesn't collide with Players group), sets NetworkOwner to the
  holding player, then fires GrabConfirmed back to client. Client then drives the
  held part's position via AssemblyLinearVelocity each Heartbeat.

  On throw: client fires RequestThrow(direction). Server applies velocity in that
  direction, reclaims NetworkOwner, resets collision group.

  An ancestry watcher auto-cleans if the held part is destroyed while held.

To make any part/model grabbable: add the CollectionService tag "Grabbable" to it.
Models are grabbed by resolving up from the hit part to the tagged ancestor.

--------------------------------------------------------------------------------
  B3. REALISTIC UPHOLSTERY WORKFLOW
--------------------------------------------------------------------------------

The real-world upholstery repair sequence that the concept game is based on:

  Step 1 — TACK REMOVAL: Strip the old cover by removing tacks/staples that hold
            the old fabric to the frame. This exposes the foam underneath.
  Step 2 — FOAM PADDING: If foam is damaged/worn, replace it. Cut and press new
            foam onto the frame/base.
  Step 3 — FABRIC DRAPING: Lay new fabric/leather over the foam, pulling it taut.
  Step 4 — SEAM FINISHING: Either stitch the fabric at the seams (needle + thread)
            or staple it down to the frame (staple gun). Some jobs use one, some
            the other.

The game uses this exact sequence as its core loop, with each step being either a
physical interaction or a GUI minigame.

--------------------------------------------------------------------------------
  B4. THE TOOLBOX SYSTEM
--------------------------------------------------------------------------------

Each player spawns with a toolbox — a physical Model in the world with the
"Grabbable" tag. The toolbox is how materials and tools are transported to jobs.

CARRYING THE TOOLBOX:
  The player grabs the toolbox with E and carries it to the job site. This is
  intentionally risky — the physics system means you could drop it, it could roll,
  etc. Future: when vehicles are added, the toolbox goes in the truck bed and
  could fall out on bumpy roads.

UNPACKING THE TOOLBOX:
  When at the job site, the player interacts with the toolbox (ProximityPrompt, no
  tool required). The toolbox Model swaps to an "open" version.

  On unpack, the game spawns the correct materials based on the job:
    - N foam pieces (N = number of sofas in the job)
    - N cloth/fabric pieces
    - 1 tack remover tool
    - 1 finishing tool: either a needle (→ seam stitching) OR a staple gun
      (→ staple gunning). This is determined 50/50 randomly when the job is
      generated. The tool in the toolbox tells the player which finishing method
      this job requires before they start.

  Foam pieces, cloth pieces, needle, and staple gun all have the "Grabbable" tag
  so the player can pick them up individually from the open toolbox.

TOOLBOX DESIGN RULES:
  - The toolbox communicates the job type (stitch vs staple) just by being opened.
  - All materials for the entire job are present from the start — no mid-job fetching.
  - The player decides in what order to use the materials across multiple sofas.

--------------------------------------------------------------------------------
  B5. STEP-BY-STEP GAMEPLAY LOOP
--------------------------------------------------------------------------------

ARRIVAL + NPC DOOR INTERACTION (see B5a for full detail):
  Player approaches the front door of the client location carrying their toolbox.
  Player knocks on the door (ProximityPrompt).
  NPC peeks out, plays invite animation, door opens.
  Player enters. Job timer starts the moment the player crosses the threshold.
  Job site has N sofas in their damaged state (SofaDamaged model).

STEP 1 — TACK REMOVAL (GUI minigame):
  The player picks up the tack remover from the open toolbox.
  ProximityPrompts appear on each sofa ONLY while the player is holding the tack
  remover (tool held = prompt visible).
  Player activates prompt on a sofa → GUI tack removal minigame plays (same
  osu!-style mechanic as the original game — reuse the existing TackRemoval.luau).
  On completion: sofa state advances to "foam-ready" (model swap if modelled).
  Player repeats for each sofa.

  IMPORTANT: Step 2 is locked until tack removal is done on that sofa.

STEP 2 — FOAM PADDING (physical):
  Player grabs a foam piece from the toolbox (it has "Grabbable" tag).
  Player carries or THROWS the foam toward a sofa that has had tacks removed.
  When the foam model touches the sofa's cushion zone (see B7 for snap detection),
  it snaps into place — the foam is now locked to the sofa.
  Snapping is immediate on contact. No minigame. Pure spatial skill.

  Physics bonus: The player can throw foam from across the room for a trickshot.
  If a thrown cloth or other object hits placed foam hard enough, it can knock it
  off — the foam un-snaps and becomes a loose Grabbable again. This is a silly
  emergent mechanic, not a core mechanic, but fun when it happens.

  IMPORTANT: Step 3 (cloth) is locked on this sofa until foam is snapped.

STEP 3 — CLOTH DRAPING (physical):
  Player grabs a cloth piece from the toolbox.
  Player carries or throws cloth toward a sofa that already has foam snapped.
  Cloth snaps on contact with the sofa (same touch detection as foam).

  MULTI-SOFA FREEDOM: The player is NOT required to finish one sofa before
  starting another. They can: foam all sofas first, then cloth all sofas. Or do
  them one at a time. Any order is fine, as long as the within-sofa sequence
  (tacks → foam → cloth → finish) is respected.

  IMPORTANT: Step 4 is locked until cloth is snapped on that sofa.

STEP 4 — SEAM FINISHING (GUI minigame):
  Determined 50/50 at job generation. Either:

  OPTION A — SEAM STITCHING:
    Player picks up the needle from the toolbox.
    ProximityPrompt appears on cloth-covered sofas only while needle is held.
    Player activates → GUI seam stitching minigame plays (reuse SeamStitching.luau).
    Same mechanic: drag needle along dotted path in one continuous motion.
    No checkpoints, graded by miss budget.

  OPTION B — STAPLE GUNNING:
    Player picks up the staple gun from the toolbox.
    ProximityPrompt appears on cloth-covered sofas only while staple gun is held.
    Player activates → GUI staple gunning minigame plays (reuse StapleGunning.luau).
    Moving bar + fire button timing mechanic.

  On completion: sofa is fully repaired. Model swap to the clean/repaired version
  (when models are available).

JOB COMPLETE:
  When all sofas are fully repaired, player exits via the door.
  Result submitted. Graded by total time taken (see B6).

--------------------------------------------------------------------------------
  B5a. NPC DOOR INTERACTION (DEMO FEATURE)
--------------------------------------------------------------------------------

This feature is intentionally demo/concept-only. It adds immersion for showcasing
the game but is too complex to scale to a full production game with many locations
and NPCs. It will NOT carry forward to the main UPHOLSTERY! game as-is.

THE FLOW:
  1. Player approaches the front door of the client house.
  2. ProximityPrompt "Knock" appears on the door (no tool required).
  3. Player activates → a knock sound plays.
  4. After a short delay (0.5–1s), the door opens slightly (~20-30 degrees).
  5. The NPC plays their "peek out" animation — leaning forward, looking at player.
     (Animation to be provided/made by the developer — custom R15 or R6 anim.)
  6. After ~1.5s of peek, NPC plays an "invite in" gesture (wave, step aside, etc.)
  7. Door opens fully. NPC steps back inside.
  8. Player can now walk through the door into the repair room.
  9. A trigger zone just inside the door starts the job timer the moment the
     player's character crosses it (Region3 or a transparent BasePart with Touched).

ROBLOX IMPLEMENTATION NOTES:
  - NPC is a Model with a Humanoid and Animator.
  - Animations are loaded via Animation instances (AnimationId set by developer).
  - The knock ProximityPrompt is on the door BasePart (or a hidden part near it).
  - Door rotation: use a HingeConstraint with ActuatorType = Motor, or simply
    tween the door BasePart's CFrame for simplicity in the prototype.
  - The NPC does NOT move from their position — only plays upper-body animations
    while standing in the doorway. No pathfinding needed.
  - Server handles the interaction (client fires remote → server plays animation,
    opens door). This is a one-shot sequence per job, not repeatable.

IMPORTANT: The NPC has no dialogue text by default — this is a pure animation
interaction. If the developer wants to add dialogue (a TextLabel above the NPC's
head), that is optional and can be added later.

--------------------------------------------------------------------------------
  B6. GRADING SYSTEM
--------------------------------------------------------------------------------

There is NO visible timer. The player never sees a countdown.
The game silently clocks the total time from job arrival to job completion.

Each job has a PAR TIME based on tier and furniture count:
  par_time = furniture_count × seconds_per_piece_for_tier

Seconds per piece by tier (approximate starting values):
  Starter:   60 seconds per sofa
  Mid:       50 seconds per sofa
  Premium:   40 seconds per sofa
  Prestige:  35 seconds per sofa
  Exclusive: 30 seconds per sofa
  Elite:     25 seconds per sofa
  Special:   20 seconds per sofa

Grading thresholds (elapsed vs par):
  Perfect: finished in ≤ 80% of par time
  Good:    finished in ≤ 100% of par time
  Pass:    finished in ≤ 140% of par time
  Fail:    took longer than 140% of par time (or abandoned)

These thresholds and par times are starting values and should be tuned after
playtesting. The key design intent: a player who works quickly and efficiently
(throws instead of walks, batches multi-sofa steps) will naturally get Perfect
without feeling rushed. A player who takes their time will still Pass.

There is no automatic failure for running out of time. The game just records
how long it took and grades accordingly. This removes frustration while keeping
the incentive to work efficiently.

--------------------------------------------------------------------------------
  B7. SNAP DETECTION — FOAM AND CLOTH PLACEMENT
--------------------------------------------------------------------------------

When a foam or cloth piece touches a valid sofa surface, it snaps immediately.

WHAT COUNTS AS "THE SOFA" (snap zone):
  The snappable zone is the cushioned body of the sofa — NOT the legs/frame.
  In Roblox terms, sofa models are typically structured as:
    SofaModel (Model)
      └── Frame (BasePart) — legs, wooden structure
      └── Cushion (BasePart or Model) — the padded area ← SNAP ZONE
      └── Tears (BasePart) — visible damage, present on Damaged variants

  Touch detection should check if the foam/cloth model intersects the Cushion part
  specifically, not the whole model. For Easy difficulty, you can widen this to
  accept any part of the sofa.

SNAP BEHAVIOR:
  On valid touch:
    1. Foam/cloth's Anchored = true (frozen in place)
    2. Weld or PivotTo to align cleanly on the sofa surface
    3. The foam/cloth part's "Grabbable" tag is removed (can't re-grab a placed piece)
    4. Sofa's step state advances (foam_placed = true, or cloth_placed = true)
    5. ProximityPrompts for the next step become available on this sofa

KNOCK-OFF (emergent mechanic, low priority):
  If another physics object hits the snapped foam/cloth with enough velocity,
  it could un-snap: Anchored = false, "Grabbable" tag restored, step state reverts.
  Player must replace it. This is a silly fun mechanic — don't make it the default
  behavior, make the threshold high enough that it only happens from real throws.

IMPLEMENTATION APPROACH (when building):
  Use Touched events on the Cushion BasePart, filtered by checking if the
  touching part belongs to a foam or cloth model that is currently being held
  (or was just released). Check that the sofa is in the correct state
  (tacks removed = true) before accepting the snap.

--------------------------------------------------------------------------------
  B8. SOFA STATE MACHINE (PER SOFA)
--------------------------------------------------------------------------------

Each sofa tracks its own repair state independently:

  State 0: DAMAGED       — starting state, tack remover prompt available
  State 1: STRIPPED      — tacks removed, foam pieces can now snap to this sofa
  State 2: FOAM_PLACED   — foam snapped, cloth pieces can now snap to this sofa
  State 3: CLOTH_PLACED  — cloth snapped, finishing tool prompt now available
  State 4: COMPLETE      — finishing minigame done, sofa is repaired

ProximityPrompts on each sofa are gated by state:
  - Tack remover prompt: visible when state == 0 AND player holds tack remover
  - Foam snap: accepted when state == 1
  - Cloth snap: accepted when state == 2
  - Finish prompt: visible when state == 3 AND player holds needle or staple gun

The server must own this state to prevent exploits.

MODEL SWAPS:
  State 0: SofaDamaged is displayed (already in place)
  State 1→3: Intermediate states (Stripped, Foam Only, Fabric Draped) — models
             not yet built. For prototype, sofa stays as SofaDamaged visually
             until State 4. Build the state machine to support swaps; add models later.
  State 4: SofaDamaged is destroyed, SofaClean is cloned in its place.
           SofaClean already exists — this model swap CAN be implemented now.

--------------------------------------------------------------------------------
  B9. TOOL MECHANICS SUMMARY
--------------------------------------------------------------------------------

  TACK REMOVER (physical item + GUI trigger)
    Grabbed with E like any object.
    While held: ProximityPrompts appear on State 0 sofas.
    On prompt activation: TackRemoval GUI minigame plays.
    After minigame: sofa → State 1. Tool stays in hand (can do next sofa).

  FOAM PIECE (physical item, snap mechanic)
    Grabbed with E. Carried or thrown toward sofas.
    Snaps on contact with Cushion zone of a State 1 sofa.
    After snap: sofa → State 2. Foam locked in place.

  CLOTH/FABRIC PIECE (physical item, snap mechanic)
    Same as foam. Snaps to State 2 sofas only.
    After snap: sofa → State 3.

  NEEDLE (physical item + GUI trigger)
    Grabbed with E. Only present if job rolled "stitching" (50/50).
    While held: ProximityPrompts appear on State 3 sofas.
    On prompt activation: SeamStitching GUI minigame plays.
    After minigame: sofa → State 4. Sofa complete.

  STAPLE GUN (physical item + GUI trigger)
    Same as needle but triggers StapleGunning GUI minigame.
    Only present if job rolled "stapling" (50/50).

--------------------------------------------------------------------------------
  B10. JOB STRUCTURE
--------------------------------------------------------------------------------

Jobs have:
  - tier: determines par time per piece and pay
  - furniture_count: 1-3 sofas (same tier scaling as original game)
  - finish_type: "stitch" or "staple" (50/50 random at generation)
  - pay: coins (same economy as original game)
  - followers: social reward
  - name: job name shown on job board before accepting

The job board and economy system can be reused directly from the original game.
The only change to job generation: add finish_type field.

--------------------------------------------------------------------------------
  B11. MULTI-SOFA FREEFORM RULES
--------------------------------------------------------------------------------

The player can work on sofas in any order, subject to per-sofa state locks.

Example with 3 sofas (A, B, C):

  VALID sequence:
    Tack remove A → Tack remove B → Tack remove C →
    Foam A → Foam B → Foam C →
    Cloth A → Cloth B → Cloth C →
    Stitch A → Stitch B → Stitch C

  ALSO VALID:
    Tack remove A → Foam A → Cloth A →
    Tack remove B → Tack remove C → Foam B → Foam C →
    Cloth B → Stitch A → Cloth C → Stitch B → Stitch C

  INVALID (state lock prevents this):
    Cloth A (before foam A is placed) — cloth snap rejected
    Stitch A (before cloth A is placed) — prompt not visible
    Foam A (before tack removal on A) — foam snap rejected

There is no system-enforced order ACROSS sofas, only WITHIN each sofa.
This design lets experienced players find efficient batching patterns, which
naturally rewards speed without making the game feel like a checklist.

--------------------------------------------------------------------------------
  B12. WHAT TO REUSE FROM THE ORIGINAL GAME
--------------------------------------------------------------------------------

These systems from the original UPHOLSTERY! codebase can be ported or reused
with minimal changes:

--------------------------------------------------------------------------------
  B14. FUTURE IDEAS (NOT FOR PROTOTYPE)
--------------------------------------------------------------------------------

  VEHICLES:
    Players drive to customer houses rather than teleporting. A pickup truck could
    carry the toolbox in the bed — bumpy roads could knock the toolbox around.
    This requires a map with actual roads between player house and client locations.
    This is a significant scope addition — do not attempt in the prototype.

  SOFA STATE MODEL SWAPS:
    Each sofa has 5 model variants (States 0-4). Damaged, Stripped, Foam Only,
    Fabric Draped, Repaired. These need to be modelled per furniture type.
    Design the state machine to support swaps now; add models later.

  DIFFICULTY SCALING:
    Easy: larger snap zone (whole sofa), more forgiving GUI minigame params
    Hard: snap zone limited to exact cushion area, tighter GUI params
    This maps directly to GameConfig.DifficultyPresets — reuse that system.

  MULTIPLE DAMAGE TYPES PER SOFA:
    A sofa could need tack removal + foam + stitch AND have a staple zone elsewhere.
    Keep this possibility in mind when designing the state machine, but don't build
    it for the prototype.

  CO-OP (EXPLICITLY OUT OF SCOPE):
    The concept is solo-only. One player, one toolbox, one job at a time.
    The grabbing system already prevents two players holding the same object.
